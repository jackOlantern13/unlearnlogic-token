Protocol Identity

UNLEARNLOGIC is not a speculation vehicle. It is a protocol designed around a single thesis: that computational participation should be the only legitimate claim to token ownership. By modeling Bitcoin's proof-of-work scarcity on Solana's high-throughput infrastructure, UNL creates a mining system that rewards sustained engagement over capital advantage.

The artistic layer is not decoration. It is the load-bearing wall. The glitch aesthetic, the terminal philosophy, the ritual of mining — these are signals in a noise-saturated economy. This is a protocol for people who build things, question things, and distrust easy money.

Core Principles

Fair Launch Only

Zero pre-mine. Zero insider allocation. Zero VC rounds. The creator treasury is hard-capped at 5% and vests linearly over 24 months via on-chain schedule. Anyone who mines before day 1 mines the same rate as anyone who mines on day 1,000.

Computational Scarcity

Token emission requires real off-chain hash computation. The protocol validates SHA-256 proofs on-chain. Minting is impossible without work. This is not a staking wrapper. This is not a yield farm. This is mining.

Authority Revocation

On Day 30 post-launch: mint authority revoked. Freeze authority revoked. Upgrade authority transferred to community multisig. The protocol becomes permanently trustless. No admin keys. No pause switches. No rescue operations.

Anti-Extractive Design

No MEV extraction from mining submission ordering. Epoch batching means no gas wars. Delayed claim windows prevent flash-bot reward sniping. The game theory rewards patience, not speed or capital.

Cultural Position

UNLEARNLOGIC exists at the intersection of cypherpunk ethics, net art, and protocol theory. It draws from the Bitcoin whitepaper's elegance, the Ethereum ecosystem's composability thinking, and the anti-algorithmic philosophies of early internet art. The name itself is the thesis: unlearning extractive financial logic as a prerequisite for participation.

The mining ritual — running a program that does real computation, waiting epochs, claiming delayed rewards — is intentionally slow. Slow is honest. Slow filters for people who believe in the protocol, not people who found it on a trending list.

Protocol Architecture

A layered system of Solana programs, PDAs, and epoch-based state machines that implement a verifiable proof-of-work mining protocol entirely on-chain, with minimal trusted infrastructure.

System Overview

Layer 0: SPL Token

Standard SPL token with mint_authority held by the Mining Program PDA initially. freeze_authority is null from genesis. Token metadata via Metaplex.

Layer 1: Mining Program

Anchor program implementing PoW proof validation, epoch management, difficulty adjustment, reward calculation, and claim settlement.

Layer 2: Treasury Program

Separate Anchor program managing creator vesting, ecosystem grants, and multisig-controlled disbursements via Squads.

Layer 3: Governance Program

Post-launch DAO transition. Initially a 3-of-5 Squads multisig. Migrates to SPL Governance at community threshold (≥1000 holders, ≥6 months).

Transaction Flows

Mining flow:

// 1. Off-chain: miner computes nonce locally
hash = SHA256(wallet_pubkey || epoch_id || nonce || challenge_seed)
// 2. Validate difficulty target met
assert hash.leading_zeros() >= current_difficulty

// 3. Submit proof on-chain (1 tx)
submit_proof({
  nonce: u64,
  hash: [u8; 32],
  epoch_pda: Pubkey,
  miner_state_pda: Pubkey,
})

// 4. Program validates:
//    - cooldown window passed (anti-spam)
//    - hash recomputed on-chain matches
//    - leading zeros >= difficulty
//    - nonce not previously used (replay protection)
//    - epoch still open
//    - updates MinerState, EpochState

// 5. After epoch close: calculate_epoch_rewards (1 tx)
// 6. After claim window: claim_rewards (1 tx per wallet)

Epoch State Machine

States: OPEN → SETTLING → CLAIMABLE → EXPIRED

OPEN       (slots 0 → EPOCH_LENGTH)
           Submissions accepted, difficulty enforced

SETTLING   (slots EPOCH_LENGTH → EPOCH_LENGTH + SETTLE_WINDOW)
           No new submissions. Reward shares calculated.
           Challenge seed for next epoch derived from
           final slot hash XOR submission merkle root.

CLAIMABLE  (slots SETTLE_WINDOW → SETTLE_WINDOW + CLAIM_WINDOW)
           Miners call claim_rewards() to receive tokens.
           Unclaimed rewards roll into ecosystem treasury.

EXPIRED    Any unclaimed after CLAIM_WINDOW
           Permanently inaccessible. Deflationary effect.

submit_proof Instruction (Core Logic)

pub fn submit_proof(
    ctx: Context<SubmitProof>,
    nonce: u64,
    hash_bytes: [u8; 32],
) -> Result<()> {
    // 1. Check paused
    require!(!config.paused, ErrorCode::ProtocolPaused);

    // 2. Epoch open
    let slot = Clock::get()?.slot;
    require!(slot < epoch.end_slot, ErrorCode::EpochClosed);

    // 3. Cooldown check (Sybil/spam resistance)
    require!(
        slot >= miner.cooldown_until,
        ErrorCode::CooldownActive
    );

    // 4. Replay protection: nonce must be > last_nonce
    require!(nonce > miner.last_nonce, ErrorCode::NonceReplay);

    // 5. Reconstruct hash on-chain (CRITICAL: no trust in client)
    let mut data = Vec::with_capacity(80);
    data.extend_from_slice(ctx.accounts.miner.key.as_ref());
    data.extend_from_slice(&config.current_epoch.to_le_bytes());
    data.extend_from_slice(&nonce.to_le_bytes());
    data.extend_from_slice(&epoch.challenge_seed);
    let computed = solana_program::hash::hash(&data);
    require!(computed.to_bytes() == hash_bytes, ErrorCode::HashMismatch);

    // 6. Difficulty check: count leading zero bits
    let zeros = count_leading_zero_bits(&hash_bytes);
    require!(zeros >= config.current_difficulty, ErrorCode::InsufficientWork);

    // 7. Update state
    miner.last_nonce = nonce;
    miner.cooldown_until = slot + config.cooldown_slots;
    miner.last_submission_slot = slot;
    epoch.total_submissions += 1;
    // Store hash quality score for reward share calculation
    submission.best_score = zeros as u64;
    submission.submitted_at = slot;

    Ok(())
}

Reward Calculation

// Epoch reward pool after halving
halving_n = current_epoch / HALVING_INTERVAL
epoch_pool = BASE_REWARD * (1/2)^halving_n

// Per-miner share: weighted by proof quality (leading zeros)
// Uses quadratic scoring to reduce winner-take-all dynamics
miner_score = miner.best_zeros^2
total_score = sum(all valid miner scores in epoch)
miner_reward = epoch_pool * (miner_score / total_score)

// Max supply safety
if config.total_minted + miner_reward > config.max_supply {
    miner_reward = config.max_supply - config.total_minted
}

Difficulty Adjustment (LWMA)

Linear Weighted Moving Average algorithm. Faster response than Bitcoin's 2016-block window. Targets a fixed number of valid submissions per epoch.

// Linearly Weighted Moving Average — 8-epoch window
TARGET_SUBMISSIONS_PER_EPOCH = 144

fn adjust_difficulty(window: [EpochStats; 8]) -> u8 {
    let weighted_sum: f64 = window.iter().enumerate()
        .map(|(i, e)| (i+1) as f64 * e.valid_submissions as f64)
        .sum();
    let weight_total: f64 = (1..=8).sum::<u64>() as f64;
    let lwma = weighted_sum / weight_total;

    let ratio = TARGET / lwma;
    // Clamp adjustment to ±2 bits per epoch
    let delta = (ratio.ln() / 2.0f64.ln()).round().clamp(-2.0, 2.0);
    (current_difficulty as i8 + delta as i8).clamp(16, 40) as u8
}

// Minimum difficulty: 16 bits (~65536 expected hashes)
// Maximum difficulty: 40 bits (extremely competitive)
// Starting difficulty: 20 bits (~1M expected hashes)

Anti-Bot Design

Dynamic Challenge Seed

Each epoch's valid challenge seed is derived from XOR(last_slot_hash, submission_merkle_root). Unpredictable until epoch opens. Cannot be pre-computed by bots before epoch start.

Per-Wallet Cooldown

Minimum cooldown_slots (default: 900, ~6 min) between any two submissions from the same wallet. Enforced in MinerState.cooldown_until.

Submission Cap

Maximum submission_cap valid submissions accepted per epoch globally. After cap reached, epoch enters early settlement. Prevents flooding with low-cost proof spam.

Quadratic Share Weighting

Reward shares weighted by score² (score = leading zero count above minimum). Better proofs earn more, but marginal returns diminish. GPU farms have diminishing advantage over efficient CPUs.

Tokenomics

Fixed supply. No inflation. Deflationary by design. Unclaimed epoch rewards are permanently burned (not redistributed), accelerating scarcity over time.

  • No token burn function intentionally — supply reduction happens through inaction, not destruction

Security Architecture

Multi-layered defense covering replay attacks, Sybil resistance, economic exploits, and program-level vulnerabilities. Pre-launch audit required.

Audit Checklist

  • All account discriminators verified in Anchor constraints
  • PDA bump seeds stored and verified on every instruction
  • No raw system_program::transfer — only SPL token CPI
  • No instruction data that exceeds compute budget
  • Clock::get() used (not deprecated sysvar_clock)
  • All arithmetic: checked_add, checked_mul, saturating operations
  • Reentrancy: no CPI back to self. All state mutations before external CPIs
  • No uninitialized memory access patterns
  • Event logging on all state-changing instructions (for off-chain indexing)
  • Fuzz testing with honggfuzz-rs on all instruction handlers
  • Invariant: total_minted <= max_supply enforced at claim time

// miner-worker.ts — runs in Web Worker thread
self.onmessage = ({ data: { wallet, epoch, difficulty, seed } }) => {
    let nonce = 0n;
    let hashRate = 0;
    const start = performance.now();

    while (running) {
        const input = encode(wallet, epoch, nonce, seed);
        const hash = sha256(input);  // wasm-crypto
        const zeros = countLeadingZeroBits(hash);

        if (zeros >= difficulty) {
            postMessage({ type: 'PROOF_FOUND', nonce, hash });
            break;
        }

        nonce++;
        // Report hashrate every 10k attempts
        if (nonce % 10000n === 0n) {
            hashRate = 10000 / ((performance.now() - start) / 1000);
            postMessage({ type: 'STATS', hashRate, nonce });
        }
    }
};

Governance Model

Progressive decentralization. Starts with a trusted multisig, ends with on-chain governance. The goal is to make the founders irrelevant.

Phase 1: Multisig (Launch → Month 6)

3-of-5 Squads multisig. Signers are publicly identified community members. Powers: emergency pause, parameter adjustment (within bounds), ecosystem grant approval. Cannot: mint tokens, override mining logic, change max supply.

Phase 2: Hybrid (Month 6 → Year 1)

SPL Governance deployed. UNL holders vote on proposals. Multisig retains veto on security-critical actions with 48h timelock. Any parameter change requires 72h on-chain vote with ≥5% quorum of circulating supply.

Phase 3: Full DAO (Year 1+)

Multisig veto power removed. All program changes require governance votes. Upgrade authority burned or held in governance-controlled PDA. Creator treasury fully vested, creator retires from governance role.

Ecosystem Treasury Governance

  • Grants require formal proposal + community vote
  • Max single grant: 50,000 UNL without super-majority
  • All disbursements published on-chain with linked proposal
  • Treasury held in Squads multisig, migrates to governance-controlled PDA
